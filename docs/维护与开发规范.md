# 维护与开发规范

## 项目边界

- 后端入口：`cmd/pxe`。
- 服务编排：`internal/app`。
- 管理 API 和前端托管：`internal/web`。
- DHCP/ProxyDHCP：`internal/dhcp`。
- TFTP：`internal/tftp`。
- HTTP Boot：`internal/httpboot`。
- 启动 URL 拼接：`internal/booturl`。
- iPXE 脚本：`internal/ipxe`。
- 数据库和模型：`internal/storage`。
- 前端源码：`web/src`。

## 修改规范

- 修改协议逻辑时同步更新日志和测试。
- 修改配置字段时同步更新默认值、校验、前端表单和文档。
- 修改 API 时保持统一响应结构，错误提示使用中文。
- API 响应中的列表字段必须返回空数组 `[]`，不能返回 `null`；Go 代码中列表结果使用 `make([]T, 0)` 或 `[]T{}` 初始化。
- 前端使用接口列表字段前必须做 `Array.isArray` 兜底，禁止直接假设后端数组一定非空非 `null` 后再读取 `.length`。
- 修改文件路径逻辑时必须保持根目录限制，禁止路径穿越。
- 修改下发给客户端的 HTTP Boot 地址时必须复用 `internal/booturl`，避免端口、通告 IP 和监听地址处理不一致。
- 修改前端页面时保持现有 Tailwind/shadcn 风格，不引入另一套设计语言。
- 不提交运行时数据、数据库、日志、node_modules、dist 产物。

## 质量检查

提交前至少运行：

```bash
go test ./...
go vet ./...
npm run typecheck --prefix web
npm run build --prefix web
```

`npm run build --prefix web` 会直接输出到 `internal/web/dist`，Go 构建会把该目录嵌入二进制。
生产二进制使用 `go build -trimpath -ldflags="-s -w"`，避免携带本地路径、符号表和调试信息。

## 发布规范

- 通过 GitHub Actions 手动触发发布。
- 版本号使用 `vMAJOR.MINOR.PATCH`。
- Release 产物包含二进制和 README。
- 发布前确认 `ai.md`、README 和 docs 与当前实现一致。

## 已知问题

- DHCP/ProxyDHCP 在不同固件、虚拟机和交换机下兼容性差异明显。
- 完整 DHCP 可能影响同网段真实设备，只能在明确网络边界后启用。
- TFTP 是 UDP 协议，丢包、MTU、blksize 都会影响稳定性。
- 外部下载能力必须提示来源 URL 和保存路径。
- 仪表盘和日志页必须共用 `web/src/lib/eventLog.ts`，避免重复 SSE、乱序刷新和无限日志堆积。
