# 使用文档

## 快速开始

1. 启动 `pxe`，打开 Web 管理页面。
2. 首次进入时创建管理员账号。
3. 在 `服务配置` 中确认 `通告 IP` 是客户端可访问的本机网卡 IP。
4. 在 `netboot.xyz` 页面下载启动文件；程序会保存到 `data/boot/netboot`，并按需生成 `data/boot/tftp/local-vars.ipxe`。
5. 在仪表盘启动服务。
6. 客户端 BIOS/UEFI 中启用 PXE 网络启动。

默认推荐使用 ProxyDHCP 模式：由现有路由器或 DHCP 服务分配 IP，本程序只提供启动信息。完整 DHCP 模式会分配 IP，只建议在隔离网络中使用。

## 启动链路

常见链路：

```text
PXE 客户端
-> 现有 DHCP 或本程序完整 DHCP 获取 IP
-> 本程序 DHCP/ProxyDHCP 返回 next-server 与启动文件
-> TFTP 下载 netboot.xyz 或 iPXE 可执行文件
-> iPXE 通过 HTTP Boot 读取 dynamic.ipxe
-> 进入 Web UI 配置的 iPXE 菜单
```

UEFI 原生菜单使用 PXE Option 43；iPXE 菜单由 `/dynamic.ipxe` 动态生成。菜单中的随机等待时间会在服务端生成实际 timeout，并应用到 UEFI 菜单和 iPXE 菜单。

## 客户端管理

`客户端` 页面支持：

- 手动添加客户端 IP/MAC。
- 批量创建待认领客户端。
- iPXE 菜单中的设备认领会把 MAC 绑定到指定 IP。
- Wake-on-LAN 唤醒。

Wake-on-LAN 会发送标准 magic packet。为提高兼容性，程序会同时向 UDP 9 和 UDP 7 发送，并覆盖全局广播、客户端单播、按客户端 IP/子网掩码推导的定向广播、本机网卡广播。实际能否唤醒还取决于设备 BIOS/UEFI、网卡电源管理、交换机/路由器是否允许广播或定向广播。

高兼容性的客户端维护能力主要包括：

- Wake-on-LAN：适合关机但网卡仍供电的设备。
- Ping 检查：适合判断设备当前网络可达性。
- HTTP 检查：适合检测客户端临时服务、安装器或健康上报端口。
- 健康报告：客户端主动向 HTTP Boot 的 `/client/report` 上报磁盘和网络信息。

## 操作菜单

`操作菜单` 页面用于维护在服务器端执行的命令模板。命令不经过 shell，变量会替换为客户端信息：

```text
%IP%
%MAC%
%NAME%
%STATUS%
%FIRMWARE%
%DISKHEALTH%
%NETSPEED%
```

内置模板由后端按服务器操作系统生成。例如 Windows 使用 `ping -n 1 %IP%`，Linux/macOS 使用 `ping -c 1 %IP%`。命令输出会按 UTF-8 解码；Windows 上遇到 GBK/GB18030 输出时会自动转换，避免中文乱码。

新增操作需要先保存，保存后后端会返回数据库中的最新 ID，随后即可选择客户端执行。

## 文件管理

`文件管理` 支持三个根目录：

- HTTP Boot：`data/boot/http`
- TFTP：`data/boot/tftp`
- netboot.xyz：`data/boot/netboot`

支持目录浏览、上传、新建目录、新建文本文件、重命名、删除、HTTP 文件制作种子、小型 UTF-8 文本在线编辑。在线编辑限制为白名单文本扩展名和 1 MiB 以内文件；镜像、固件、压缩包不会读入浏览器。

## netboot.xyz

`netboot.xyz` 页面会显示启动文件是否已下载、保存路径、文件大小、更新时间和下载结果。点击下载后：

- 已存在且非空的文件会跳过。
- 下载完成会显示 SHA256。
- `local-vars.ipxe` 不存在时自动生成，存在时不会覆盖。

本地钩子提供 Debian 12 和 Alpine Linux 的公网/内网启动菜单。内网路径会根据当前通告 IP 和 HTTP Boot 端口生成，例如：

```text
http://通告IP/debian/dists/bookworm/main/installer-amd64/current/images/netboot/debian-installer/amd64/linux
http://通告IP:8080/alpine/v3.23/releases/x86_64/netboot/vmlinuz-lts
```

你需要把对应内核和 initrd 放到 `data/boot/http` 下的相同路径中。

## 自定义 iPXE 固件

仓库提供 `embed.ipxe` 和 `.github/workflows/build-boot.yml`，用于构建带内置菜单的 iPXE 固件。当前内置菜单会直接访问公网镜像，提供 Debian 12、Alpine Linux、iPXE Shell 和退出项。

构建出的常见产物：

- `undionly.kpxe`：Legacy BIOS 常用。
- `ipxe-x86_64.efi`：现代 UEFI x86_64 常用。
- `ipxe-arm64.efi`：UEFI ARM64 常用。

这些文件需要人工放到运行时启动目录或按配置使用。当前 DHCP 启动文件优先级是：

- BIOS：优先 `data/boot/netboot/netboot.xyz.kpxe`，其次 `data/boot/netboot/netboot.xyz-undionly.kpxe`，再回退到服务配置的 BIOS 启动文件。
- UEFI：优先 `data/boot/netboot/netboot.xyz.efi`，再按 UEFI32/UEFI64 回退到服务配置字段。

因此，如果要使用自构建固件，最稳妥的方式是在受控网络中按架构放置对应文件，并确认 DHCP 下发的文件名与客户端架构一致。ARM64 固件已经可以通过流水线构建，但当前服务配置没有独立 ARM64 EFI 字段；混合 x86_64/ARM64 UEFI 网络中不要让两类设备共用同一个 `netboot.xyz.efi` 文件名，除非你明确知道该文件与目标设备匹配。

注意区分三类脚本：

- `embed.ipxe`：编译进 iPXE 固件，修改后必须重新构建固件。
- `data/boot/tftp/local-vars.ipxe`：netboot.xyz 页面按需生成的本地钩子，已存在时不会覆盖。
- `/dynamic.ipxe`：服务运行时按数据库菜单动态生成，主要给已经进入 iPXE 的客户端使用。

## 离线安装

以 Legacy BIOS 为例：

1. 下载官方 iPXE 文件 `https://boot.ipxe.org/undionly.kpxe`，改名为 `netboot.xyz-undionly.kpxe` 后放入 `data/boot/netboot/`。
2. 将内核、initrd 和安装配置放入 `data/boot/http/`。
3. 新建 `data/boot/http/boot.ipxe`：

```ipxe
#!ipxe
set base-url http://192.168.137.1
kernel ${base-url}/linux initrd=initrd.gz ip=dhcp
initrd ${base-url}/initrd.gz
boot
```

4. 启动 PXE 客户端，进入 iPXE 菜单后选择 `Run boot.ipxe`。

完全离线时不要依赖公网 netboot.xyz 菜单，应使用本地 `boot.ipxe` 和本地镜像。

## 排障

- 客户端拿不到 IP：检查完整 DHCP 是否与路由器冲突，ProxyDHCP 模式下确认已有 DHCP 服务正常。
- TFTP 超时：检查防火墙是否放行 UDP 69、UDP 4011、UDP 67。
- HTTP 文件打不开：从客户端同网段机器访问 `http://通告IP/` 验证。
- WOL 无效：确认 BIOS/UEFI 已启用网络唤醒，系统关机后网卡仍供电，交换机没有拦截广播。
- 操作菜单无输出或失败：确认命令存在于服务器，新增操作已保存，目标客户端 IP/MAC 不为空。
